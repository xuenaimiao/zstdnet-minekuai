# 省流量进阶设置

装好之后**默认就已经在压缩了**，这一篇是给「想压得更狠」的人看的。
所有进阶开关都是**默认关闭、可选打开**：不动它们，行为和现在完全一样，也不会影响没升级的玩家。

> 改服务端：`config/zstdnet-server.properties`（插件服在 `plugins/ZstdNet/zstdnet-server.properties`）。
> 改客户端：`config/zstdnet-client.toml`。改完一般几秒内自动热重载，或重连一次即可。

一共三个值得了解的开关，按推荐顺序：

1. [字典（最推荐，一键）](#1-字典最推荐一键)
2. [实体变换 transform（实体/生物特别多的服）](#2-实体变换-transform实体生物特别多的服)
3. [压缩等级 level（性能换压缩率）](#3-压缩等级-level)

还有一个进阶的 [长距离匹配 LDM](#4-长距离匹配-ldm) 简单提一下。

---

## 1. 字典（最推荐，一键）

**它是什么**：让压缩算法先「认字」——拿一本根据你这台服流量训练出来的「字典」，
对登录时的爆发数据、海量相似小包压得更狠。

**怎么开**：服务端一个开关就够，全自动：

```properties
dictionary_auto=true
```

开了之后服务器会自己：采样在线流量 → 攒够样本自动后台训练 → 训练完自动启用 → 自动下发给玩家。
**不用重启、不用手动分发**。玩家首次进服会自动下载字典、被提示重连一次，之后就享受字典压缩了。

> 玩家端**不用做任何事**，字典会自动下载。

---

## 2. 实体变换 transform（实体/生物特别多的服）

**先说适合谁**：

- 机械动力（Create）服，尤其**契约体 / 移动机器**多、一直在动；
- **刷怪塔、怪潮、大型农场**，成片相似的怪 / 动物；
- 简单说：只要你的服「实体特别多、动得特别勤」，这个最有用。

**它解决什么**：实体的移动、转头、速度包，以及大量相似生物的数据，里面有大量重复内容，
但被「结构字节」打散了，普通压缩不好下手。transform 在压缩**之前**把这些同类数据**理顺、聚到一起**，
再压就能压得很狠，解压时**逐字节精确还原**，玩法完全不受影响。

**效果**（实测）：

- 机械动力契约体场景：相关流量能再降到原来的约 **1/8**（省 ~87%）；
- 大量相似生物场景：再省约 **27%**。

**怎么开**：这是唯一一个**两端都要开**的功能（一端理顺、另一端要会还原）：

- 服务端 `zstdnet-server.properties`：
  ```properties
  transform=true
  ```
- 客户端 `zstdnet-client.toml`：
  ```toml
  transform = true
  ```

只有**两边都为 true** 时这条连接才真正启用；任意一边没开 / 没升级，会**自动回退**到原样转发，
连接和玩法都不受影响（默认安全）。

**关于版本**：实体级的深度优化覆盖 **1.19.2 / 1.20.1 / 1.21.1**；其他版本会自动退化成「通用去交错」，
仍有一定收益。服务端还有个 `transform_max_version`（默认 3，一般不用动）。

> 放心：这个功能的正确性**不依赖任何内部数据表**——即使内部识别表有误，最坏只是「少压一点」，
> **绝不会损坏数据**。

**怎么确认有效**：进游戏开 `/zstdhud`，到实体密集的地方（契约体旁、刷怪塔），看 **Ratio** 是否比不开时更低。

---

## 3. 压缩等级 level

**它是什么**：压缩使劲程度。数字越大压得越狠，但越费 CPU。

- 服务端 `zstdnet-server.properties`：`level=9`（范围 1–22）
- 客户端 `zstdnet-client.toml`：`level = 3`

**建议**：

- 一般服务器 `3–9` 足够，是压缩率和性能的平衡点。
- CPU 紧张就调低（比如 3–5）；CPU 富裕、想极致省流量可调高。
- 注意：调高 level 主要吃 CPU，**降流量的「大头」其实是上面的字典和 transform**，建议优先用那两个。

---

## 4. 长距离匹配 LDM

**它是什么**：让压缩能「记住」几分钟前出现过的大块结构，再次出现时直接引用。适合
「同样的大结构在一段时间内反复出现」的服（Create 系 / 大整合包）。

**怎么开**（服务端）：

```properties
long_distance_matching=true
# 可选：window_log=25   （记忆窗口，越大越能匹配远处但越吃内存；≤27 时老客户端无需升级即可解码）
```

**代价**：每条连接、每个方向约 `2^window_log` 字节内存，繁忙服务器会累加，所以默认关闭。

---

## 怎么看到底省了多少？

- **玩家**：游戏内 `/zstdhud` 看 Ratio（越低越省）。
- **服主**：看服务端日志里周期打印的统计行：
  ```text
  Raw: 189.06 GB | Zstd: 10.28 GB | Ratio: 5.44% | Conns: 8
  ```
  `Raw` 是不压会传的量，`Zstd` 是实际传的量，`Ratio` 越低越好。

---

## 一句话总结该开哪个

- **几乎所有服**：`dictionary_auto=true`（一键，收益大）。
- **实体/生物特别多的服**：再加 `transform=true`（**服务端+客户端都开**）。
- CPU 富裕想再榨一点：适当调高 `level`，或开 `long_distance_matching`。
